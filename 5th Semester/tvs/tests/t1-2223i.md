## a
```
Indique, justificando, o que é afixado no standard output e qual o conteúdo dos ficheiros output*.txt, com a execução de cada uma das seguintes linhas de comandos pelo shell:

    ex1 18 | grep 1 > output1.txt ← também para a alínea b
    ex1 77 | tee output2.txt | grep 8
```

```
ex1 18 | grep 1 > output1.txt

É executado o comando ex1 com o argumento 18, redirecionando o seu stdout para o pipe e posteriormente para o stdin do comando grep 1.

Este último por sua vez tem o seu stdout redirecionado para o ficheiro output1.txt.

Portanto, nada é afixado no stdout, mas o ficheiro output1.txt contém o output do comando grep 1, que é 18, 19 e 21, uma vez que o grep filtra as linhas que contêm o número 1.

ex1 77 | tee output2.txt | grep 8

tee -> lê do stdin e escreve para o stdout e para 1 ou mais ficheiros

É executado o comando ex1 com o argumento 77, redirecionando o seu stdout para o pipe e posteriormente para o stdin do comando tee output2.txt.

Depois o stdout do comando tee é redirecionado para o pipe e posteriormente para o stdin do comando grep 8.

Portanto o output2.txt contém o output do comando ex1 77, que é 77, 78, 79 e 80, e no stdout é afixado o output do comando grep 8, que é 78 e 80, uma vez que o grep filtra as linhas que contêm o número 8.
```

## b
```
Escreva um programa em C que execute especificamente a primeira linha de comandos indicada na alínea anterior, usando diretamente a API POSIX para criar/abrir ficheiro(s), lançar os processos necessários, ligá-los da mesma forma que o shell e aguardar pela sua conclusão.
```

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
	FILE *output1_fp;
	
	int pipecp[2];
    pipe(pipecp);
    
    pid_t pid = fork();
    if (pid == 0) {
    	close(pipecp[0]);
    	dup2(pipecp[1], 1);
		close(pipecp[1]);
		
		for (int i = 0; i < 4; ++i) printf("%d\n", 18 + i);  
		exit(0);   
	}
	else { // parent
		close(pipecp[1]);
		dup2(pipecp[0], 0);
		close(pipecp[0]);
		
		output1_fp = fopen("./output1.txt", "w");
		dup2(fileno(output1_fp), 1);
		
    	char* args[] = {"grep", "1", NULL};
		execvp("grep", args);

		close(pipecp[0]);

        int status = 0;
        waitpid(pid, &status, 0);
		
		if(output1_fp != NULL) fclose(output1_fp);
	}
	return 0;
} 
```

## a
```
Qual é o endereço virtual mais baixo que poderia estar a ser traduzido? Apresente os cálculos.
```

```
Se for um endereço virtual que use 4-level paging, então:

16 bits de extensão de sinal
9 bits de PML4
9 bits de PDP
9 bits de PD
9 bits de PT
12 bits de offset

   PML4           PDP          PD           PT
0 0001 0001  0 0000 0001  0 0000 1001  0 0000 0011

De forma a ser o menor possível, os bits de PP têm de ser todos 0, e uma vez que os bits de extensão de sinal são todos 0, o endereço virtual mais baixo que poderia estar a ser traduzido é:

0000 0000 0000 0000  0000 0000 1000 0000 0100 0001 0010 0000 0011  0000 0000 0000 = 0x0000008041203000

const int BASE[] = {0,11,22,33};
int accum = 1;
int func(int idx) { accum += idx; return BASE[idx%4] + accum; }
```

## a
```
Tendo em conta apenas o código apresentado, que alterações se pode prever que terá o espaço de endereçamento de um processo que carregue esta biblioteca com dlopen?
```

```
Ao carregarmos a biblioteca com dlopen, o espaço de endereçamento do processo que a carrega, irá criar uma nova região de data inicializada (.data) e uma nova região de código (.text).
```

## b
```
Se, simultaneamente, quatro processos distintos carregarem esta biblioteca e executarem pelo menos uma vez a função func, indique, justificando, qual espera que seja o proportional set size (Pss) das regiões relevantes relativas à biblioteca logo após os retornos de func?
```

```
Uma vez que o Pss é uma porção da memória RAM ocupada por um processo e é composto por memória privada desse mesmo processo e memória partilhada com outros processos, o Pss das regiões relevantes relativas à biblioteca logo após os retornos de func será igual ao valor antes de func ser executada, uma vez que já não existe memória partilhada entre os processos.
```
